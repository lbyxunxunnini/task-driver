# Task Driver Error Templates

遇到异常时，必须使用以下结构化模板输出提示，确保用户能快速定位问题类型、原因和下一步动作。

## 停机回问模板

````markdown
⚠️ **停机：[问题类型]**

**当前阶段**：[brainstorming / planning / executing / verification]
**当前任务**：T-NNN（如有）

**问题描述**：
[一句话说明具体问题]

**上下文**：
- [相关文件/命令/输出]
- [触发条件]

**需要决策**：
[明确说明需要用户围绕同一个决策点做什么选择]

**选项**：
1. [选项A] -> [预期结果]
2. [选项B] -> [预期结果]
3. [选项C] -> [预期结果，可选]

**推荐**：
[推荐选项]，因为 [理由]
````

## 验证失败模板

````markdown
❌ **验证失败：[AC-ID / T-NNN]**

**失败项**：[验收标准描述]
**验证命令**：`[命令]`
**实际结果**：
```
[关键输出摘录]
```

**预期结果**：
[期望行为]

**证据强度**：[weak / stale / none]

**建议下一步**：
1. [具体修复方向]
2. [或] 进入 `blocked` / `partial` / `plan-revision`
````

## 循环退出模板

````markdown
🔄 **执行-验证循环已达上限**

**Requirement**：[AC-ID / 描述]
**已尝试**：2 轮

**Iteration Log**：
| attempt | hypothesis | command | result | outcome |
|---------|------------|---------|--------|---------|
| 1 | [假设] | [命令] | [结果] | failed |
| 2 | [假设] | [命令] | [结果] | failed |

**根因分析**：
[判断失败的根本原因]

**建议状态**：
- `blocked`：[原因] -> 需要用户决策：[具体问题]
- `partial`：[已完成部分] -> 缺口：[未完成部分]
- `plan-revision`：[plan 假设错误点] -> 需要回到 planning 阶段
````

## 范围漂移模板

````markdown
🛑 **Scope Drift 检测**

**当前任务**：T-NNN
**允许文件集**（PlanPacket File Map）：
- [文件1]
- [文件2]

**实际改动**：
- [文件A] ✅
- [文件B] ✅
- [文件C] ⚠️ 不在允许集合中

**超出原因**：
[说明为什么改了计划外文件]

**需要决策**：
是否批准扩展 scope？如否，需要回滚 [文件C] 的改动。
````

## 阻塞状态模板

````markdown
🔒 **状态：Blocked**

**阻塞项**：[AC-ID / T-NNN / 整体]
**阻塞原因**：[环境 / 权限 / 外部服务 / 用户决策 / 依赖]

**详情**：
[具体说明阻塞情况]

**解除条件**：
[需要什么才能继续]

**当前产出**：
- [已完成的部分]
- [可交付的部分（如有）]
````
