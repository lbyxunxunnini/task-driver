# 快速开始

## 30 秒选择器

使用任务驱动：

- 目标、范围或验收不清楚。
- 跨多个文件、模块或外部系统。
- 涉及权限、费用、发布、分支、提交、合并、删除或覆盖重要文件。
- 用户要求先计划、先澄清或完整推进。

不使用任务驱动：

- 简单问答。
- 只需要执行一个安全命令。
- 单文件低风险小改动，且用户明确要求直接改。

## 按需阅读

- 不清楚流程原因、状态、权限或重试规则：查看 [常见问题](faq.md)。
- 需要判断某类任务怎样讨论和验收：查看 [业务场景最佳实践](scenarios.md)。
- 需要完整协议：查看根目录 [SKILL.md](../SKILL.md)。

## 固定链路

```text
仔细询问并逐项讨论
-> 输出计划
-> 讨论并确认计划
-> 讨论验收目标
-> 输出并确认验收目标
-> 创建目标并执行
-> 未达标或无法确认达标时先分类
-> 可修复问题打回修复，最多 3 次；其他异常进入对应路径
```

## 每一步怎么说

### 逐项讨论

每轮只处理一个用户决策点。先说明事实、取舍和建议，再请用户决定当前项。

```text
当前先讨论产物结构。

已确认最终只需要动效、完整成片和生产记录，并且目录不能复杂。
我的建议是采用扁平结构……优点是……代价是……

请确认当前这一项。确认后再讨论素材来源。
```

不要在同一轮顺带展开素材来源、提示词、字幕和校验方法。

### 结束讨论

模型判断关键问题已经闭合后，只能提供选择，不能直接输出计划。

```text
当前关键问题已经讨论充分。

请选择：
1. 结束讨论并生成计划。
2. 继续讨论。
```

### 输出计划

顶层计划事项对应已确认的需求或问题。每项必须包含“已确认”和“实施方案”。

```markdown
# 计划草案

任务目标：修复登录按钮错位。

## 1. 修复按钮位置

### 已确认
- 只修登录按钮，不统一其他按钮样式。

### 实施方案
调整按钮所在布局的对齐约束，保留现有视觉样式和其他按钮行为。
```

复杂任务可以增加“预期变化、涉及位置、内部验证”。不要把文件、测试和版本更新拆成新的顶层事项。

### 讨论并确认计划

用户修改计划时更新同一份计划。计划确认后进入验收目标讨论，不直接输出目标草案。

### 讨论验收目标

逐项讨论“什么结果代表对应需求或问题已经完成”。

```text
先讨论按钮错位的最终结果。
我的建议是：桌面宽度下按钮保持在预期位置，其他按钮样式不变。
这是否准确？
```

模型判断目标口径闭合后，让用户选择继续讨论或生成最终验收目标。

### 输出并确认验收目标

默认使用简短编号列表，只展示必要终极结果。

```text
1. 桌面宽度下登录按钮保持在预期位置，其他按钮样式不变。
2. 移动端窄屏下按钮保持在表单容器内，不发生横向溢出。
```

不设置固定条数。合并或拆分时说明对应的原始事项和原因。测试、脚本、文档和版本默认留作内部证据。

### 创建目标并执行

用户确认最终验收目标后，原样创建目标或维护轻量目标，再连续执行。任何目标未达成都不能标记完成。

### 自检打回

按最终验收目标逐项检查。未达标或无法确认达标时先分类，只有可修复问题进入修复和复验，每项目标最多 3 次。

先分类再处理：

- 可修复：改变实施策略后修复和复验，最多 3 次。
- 需要用户验收：完成 Agent 能做的检查，再提供具体验收步骤。
- 需要用户授权或决定：立即停止相关执行，请用户选择，不消耗修复次数。
- 受阻：说明已经尝试过什么和为什么没有替代路径，不机械重试。

统一通知最小结构：

```text
{未达标 / 需要用户验收 / 需要用户授权或决定 / 受阻}

当前结果：...
原因：...
已完成：...
下一步：...
```

未达标还要说明本轮改变的策略和修复次数；需要用户操作时必须补充具体步骤和通过条件。

### 需要用户验收

Agent 无法可靠验证某项目标时，先完成自己能做的检查，再给出具体通知：

```text
需要用户验收

原因：该效果依赖真实设备上的触控体验，Agent 无法可靠判断。

验收步骤：
1. 在指定设备打开目标页面。
2. 按正常使用方式完成一次操作。
3. 重复操作三次，观察是否出现误触或卡顿。

通过条件：三次操作均完成，且没有误触、卡顿或异常跳转。

请反馈：通过或不通过；不通过时请说明出现问题的步骤和现象。
```

用户反馈前不能把该目标标记为达标。禁止只说“请自行检查”而不给步骤和通过条件。

## 典型反例

### 用完整计划代替讨论

```text
用户：开始详细讨论目录、素材和提示词。
助手：下面是完整九步计划……
```

判定：跳过逐项讨论。

### 模型建议冒充用户决定

```text
用户：目录不要复杂。
助手：默认采用 inputs/、render/、metadata/ 等分层目录……
```

判定：模型工程偏好覆盖用户约束。

### 计划只有任务名

```text
1. 修改流程。
2. 更新文档。
3. 增加测试。
```

判定：没有展示已确认决策和实施方案，也没有按用户问题组织计划。

### 验收目标扩散成施工清单

```text
1. SKILL.md 已修改。
2. 测试已增加。
3. README 已更新。
4. 版本号已同步。
```

判定：内部实施工作冒充终极目标。

## 禁止事项

- 禁止把待确认建议写成默认技术决策。
- 禁止模型自行结束讨论并输出计划或验收目标。
- 禁止计划缺少实施方案。
- 禁止按文件和测试膨胀顶层计划事项或最终验收目标。
- 禁止把辅助检查通过当成最终目标达成。
- 禁止在需要用户验收时只给笼统提醒，不提供步骤和通过条件。
- 禁止未经授权新建分支、提交、推送或合并。
