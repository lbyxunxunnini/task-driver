# 业务场景最佳实践

本文件帮助模型和用户判断不同任务应重点讨论什么。场景建议不能代替用户决定，完整流程仍以根目录 `SKILL.md` 为准。

## 低风险 Bug 修复

适用情况：问题范围明确、改动可逆、不涉及外部权限或生产操作。

重点讨论：

- 用户可见的错误表现和正确表现。
- 只修当前问题还是覆盖相邻同类问题。
- 哪些现有行为必须保持不变。

计划重点：说明根因方向、核心修复机制和影响边界，不按文件拆成施工清单。

最终验收目标：错误不再出现，相关正常路径保持原行为。

Agent 内部验证：复现问题、运行相关测试、检查回归路径和差异范围。

用户验收：通常不需要；只有 Agent 无法访问真实界面或设备时才请求。

停止或拆分条件：发现根因涉及独立模块、权限或另一个业务目标时，返回讨论并确认是否拆分。

## 新功能或新模块建设

适用情况：需要新增用户能力、接口、模块或完整产物。

重点讨论：

- 为什么需要该能力，最终使用者是谁。
- 正常路径、失败路径和明确非目标。
- 核心接口、数据、交互和兼容取舍。

计划重点：展示总体实施机制、模块边界、数据或交互流和必要集成点。

最终验收目标：用户能够完成新能力对应的完整结果；失败时有明确行为。

Agent 内部验证：端到端小案例、单元或集成测试、错误路径和兼容检查。

用户验收：涉及主观体验、真实业务数据或私有环境时需要。

停止或拆分条件：一个请求包含多个可独立交付能力，或不同能力需要独立发布和回滚时拆分。

## 跨模块重构

适用情况：目标是改善结构、迁移接口或替换内部机制，同时保持用户行为或明确改变部分行为。

重点讨论：

- 重构的业务原因和必须保持的外部行为。
- 允许改变与禁止改变的接口、数据和兼容边界。
- 迁移顺序、回滚路径和完成分母。

计划重点：说明新旧结构、迁移路径、兼容策略和每个阶段怎样保持系统可用。

最终验收目标：确认范围内完成迁移，外部行为符合约定，没有遗漏旧入口或静默降级。

Agent 内部验证：范围清单、调用链检查、迁移前后行为对比、回滚和集成测试。

用户验收：涉及真实生产流量、业务报表或外部消费者时需要。

停止或拆分条件：不同模块可独立迁移、需要不同负责人或回滚周期时，按迁移批次拆分。

## 生产配置、外部接口或费用任务

适用情况：涉及生产环境、真实 API、费用、发布、权限或不可逆操作。

重点讨论：

- 明确授权范围、费用上限和可操作环境。
- 成功、失败、降级和回滚路径。
- 凭据来源、敏感信息边界和外部服务限制。

计划重点：说明调用或发布机制、风险控制、停止条件和回滚方式。

最终验收目标：授权范围内的真实结果达成；失败时没有越权、超额费用或不可恢复状态。

Agent 内部验证：尽可能先用只读、模拟、隔离或最小真实调用验证。

用户验收：Agent 无法访问真实账号、生产结果或外部控制台时必须请求。

停止或拆分条件：缺少授权、费用确认、凭据或回滚能力时立即停止，不消耗修复次数。

## UI、设备体验或主观质量任务

适用情况：结果依赖视觉效果、触控体验、声音、真实设备表现或人的主观判断。

重点讨论：

- 目标设备、环境和主要用户路径。
- 用户认为通过与不通过的具体表现。
- Agent 可以自动验证的部分和必须人工验收的部分。

计划重点：说明实现方法、目标环境、可自动检查内容和用户验收安排。

最终验收目标：目标环境中的实际体验符合用户确认的表现，没有已知瑕疵。

Agent 内部验证：构建、静态检查、自动化测试、截图或可访问环境中的行为验证。

用户验收：给出具体设备、操作步骤、预期结果、通过条件和反馈要求；反馈前不能标记达标。

停止或拆分条件：不同设备或体验路径具有独立结果和独立修复方式时，可按环境或用户路径拆分并说明原因。
