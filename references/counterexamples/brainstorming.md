# Brainstorming Counterexamples

这些反例命中时，必须停在 brainstorming 阶段，不得进入 planning 或 implementation。

## 本地事实转嫁给用户

- 错误：没有读取 README、目录、git status、日志或相关文件，就问用户“要改哪些文件？”
- 违规：本地可查事实被转嫁给用户，spec 依据不足。
- 回退：先做本地事实收集；只把用户独占事实作为单问题澄清。

## 问卷式澄清

- 错误：一次列出目标、范围、验收、技术偏好、上线时间等多个问题。
- 违规：违反单问题澄清门，用户回答难以映射到 spec 决策。
- 回退：只问当前最高影响的一个决策点，并给 2-3 个互斥选项和推荐。

## 伪闭合 spec

- 错误：写“先按常规实现”“细节后续补”“大概支持主要场景”作为 Goal、Scope 或 AC。
- 违规：必填门禁没有闭合，验收不可执行。
- 回退：补齐 Goal、Scope、Non-goals、Acceptance Criteria、Constraints、Quality Level。

## 重新规划跳到实现细节

- 错误：用户要求整体规划或重构规划时，第一问直接问文件、类名、接口或页面细节。
- 违规：跳过整体规划视角、大类划分和范围切片。
- 回退：第一问必须确认规划视角或大类划分。

## 默认业务决策

- 错误：自动选择质量层级、验收范围、发布风险或业务优先级，并写入 spec。
- 违规：这些属于用户决策，不允许默认。
- 回退：作为单问题澄清项让用户拍板。
