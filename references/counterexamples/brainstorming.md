# Brainstorming Counterexamples

这些反例命中时，必须停在 brainstorming 阶段，不得进入 planning 或 implementation。

## 本地事实转嫁给用户

- 错误：没有读取 README、目录、git status、日志或相关文件，就问用户“要改哪些文件？”
- 违规：本地可查事实被转嫁给用户，spec 依据不足。
- 回退：先做本地事实收集；只把用户独占事实作为单问题澄清。

## 问卷式澄清

- 错误：一次列出目标、范围、验收、技术偏好、上线时间等多个问题。
- 违规：违反单问题澄清门，用户回答难以映射到 spec 决策。
- 回退：只问当前最高影响的一个决策点，并给 2-3 个互斥选项和推荐。

## 伪闭合 spec

- 错误：写“先按常规实现”“细节后续补”“大概支持主要场景”作为 Goal、Scope 或 AC。
- 违规：必填门禁没有闭合，验收不可执行。
- 回退：补齐 Goal、Scope、Non-goals、Acceptance Criteria、Constraints、Quality Level。

## 重新规划跳到实现细节

- 错误：用户要求整体规划或重构规划时，第一问直接问文件、类名、接口或页面细节。
- 违规：跳过整体规划视角、大类划分和范围切片。
- 回退：第一问必须确认规划视角或大类划分。

## 技术方案未闭合就进入计划

- 错误：用户还没确认目标、范围、验收、风险边界或取舍轴，就把多个技术方案拆成任务并准备开工。
- 违规：会改变接口、依赖、验证方式、回滚方式、用户流程或风险边界的技术方案属于 spec 决策，不是 plan 细节。
- 回退：停在 brainstorming，按宏观到细节顺序只问当前最高影响的一个方案决策，并给 2-3 个互斥选项和推荐。

## 拷问跳层

- 错误：用户只说“提升 Task Driver 质量”，agent 直接问“要不要改 SKILL.md 的某个段落”或直接列实现任务。
- 违规：跳过整体目标、大类/规划轴、范围切片和验收标准，未形成 Decision Trace。
- 回退：回到 brainstorming，从当前缺失的最高层级开始拷问；每轮只问一个决策点，但必须追到该层闭合。

## 只问主路径

- 错误：只确认正常流程怎么走，没有追问错误路径、边界条件、验证方式、风险和回滚。
- 违规：宏观到细分必须覆盖设计树分支，不只是主路径线性推进。
- 回退：建立 Design Tree Coverage，把行为、验证、风险、替代方案分支逐一标为 decided / deferred / out_of_scope；open 分支继续单问题拷问。

## 宏观问题浅闭合

- 错误：用户说“目标就是提升质量”，agent 直接认为整体目标已闭合。
- 违规：整体目标必须追到目标对象、失败现状、成功外观和可验证完成定义；笼统目标不是闭合目标。
- 回退：继续追问整体目标分支，直到 target_statement 和 success_definition 可映射到 AC。

## 同层替代路径未处理

- 错误：确认采用方案 A 后，未记录方案 B/C 为什么不做，也没有写入 rejected_paths 或 Alternatives Considered。
- 违规：方案分支必须处理主要替代路径；否则 plan 中的技术路线缺少共同理解。
- 回退：补问方案取舍轴，记录推荐、用户决策和拒绝路径。

## Design Tree Coverage 缺失

- 错误：spec 只有 Decision Trace，没有 Design Tree Coverage，无法看出哪些分支仍 open。
- 违规：Decision Trace 记录决策链，Design Tree Coverage 负责证明分支覆盖；二者缺一不可。
- 回退：补 Design Tree Coverage；若存在 open 分支，继续 brainstorming。

## 用户笼统回答后伪闭合

- 错误：用户回答“都要做好一点”，agent 直接写 spec 或 plan。
- 违规：笼统回答没有形成可执行决策，也没有说明排除项、AC 影响、风险边界和验证方式。
- 回退：把笼统回答转成下一层具体问题，例如先确认规划轴、范围切片或验收证据。

## 事后补 Decision Trace

- 错误：没有一问一答推进数据收集状态，最后直接生成一张 Decision Trace 表说已经闭合。
- 违规：Decision Trace 是数据收集过程的记录，不是事后包装；必须有 current_branch、上游依赖、推荐答案、用户决策和未闭合分支的推进过程。
- 回退：回到当前最高影响的未闭合分支，继续单问题数据收集。

## 未共享理解就批准 spec

- 错误：用户只回复“继续”或“可以”，agent 直接把 spec 标为 Approved。
- 违规：共享理解确认门要求用户确认目标、范围、非目标、关键取舍、验证方式和风险摘要；泛化继续不等于 shared_understanding。
- 回退：输出共享理解摘要，请用户确认或修正；确认前 spec 保持 Draft。

## 默认业务决策

- 错误：自动选择质量层级、验收范围、发布风险或业务优先级，并写入 spec。
- 违规：这些属于用户决策，不允许默认。
- 回退：作为单问题澄清项让用户拍板。

## 目标空泛就开工

- 错误：用户说“优化这个 skill”或“整改所有问题”，agent 直接拆 Phase 或写计划。
- 违规：没有精准目标、范围分母、目标原则、完成定义和功能级检验规则。
- 回退：先建立 Target 草案；用单问题澄清门确认目标对象、失败现状、成功外观和可计数分母。

## 百分比目标无分母

- 错误：把“100% 迁移”“全部覆盖”“完整整改”写入 spec，但没有列出模块、文件族、命令、配置、测试、文档、用户路径或阶段分母。
- 违规：无法判断 100% 的分母，也无法在 verification 阶段证明完整性。
- 回退：继续 brainstorming，确认 scope_denominator 和每个目标单元的完成判定。

## 拆解轴未确认

- 错误：从“提升 Loop 成熟度”直接跳到“Phase 1 / Phase 2 / Phase 3”，没有确认按阶段、模块、风险、用户路径、产物类型还是协议层级拆。
- 违规：拆解轴会影响 Scope、AC、Plan File Map 和验证策略，属于 spec 生成路径决策。
- 回退：回到当前最高影响的拆解轴问题，给 2-3 个互斥选项和推荐。

## 评价模型默认生成

- 错误：直接写“建立 4 维度 Rubric”或“5 平台 x 5 维度评分”，没有确认评价对象、维度来源、尺度、阈值、证据来源和失败反例。
- 违规：评价模型决定验收口径，不能由 agent 在 plan 中自行推荐后落地。
- 回退：把评价模型作为设计对象继续单问题澄清；确认前 spec 保持 Draft。
