# 任务驱动

任务驱动是一个低认知负担的复杂任务推进 skill。它先和用户逐项形成方案，再输出含实施方案的计划，共同确定最终验收目标后创建目标执行。

## 30 秒选择器

使用任务驱动：

- 目标、范围或验收不清楚。
- 涉及多个文件、模块或外部系统。
- 涉及权限、费用、发布、分支、提交、合并、删除或覆盖重要文件。
- 用户要求先计划、先澄清或完整推进。

不使用任务驱动：

- 简单问答。
- 只需要执行一个安全命令。
- 单文件低风险小改动，且用户明确要求直接改。

谨慎使用：

- 用户要求立即执行，但任务存在风险动作时，先确认授权和最终验收目标。
- 任务看似很小，但失败会影响生产、数据、费用或不可逆状态时，按任务驱动推进。

## 核心链路

```text
仔细询问并逐项讨论
-> 输出计划
-> 讨论并确认计划
-> 讨论验收目标
-> 输出并确认验收目标
-> 创建目标并执行
-> 未达标或无法确认达标时先分类
-> 可修复问题打回修复，最多 3 次；其他异常进入对应路径
```

这条链路分别回答：

- 为什么做、关键取舍是什么：逐项讨论。
- 做什么、准备怎样实施：计划。
- 做到什么结果才算完成：最终验收目标。
- 怎样持续做到目标：执行、自检和打回修复。

## 使用方式

当用户消息以以下任一形式开头时，进入任务驱动：

```text
tdr- ...
task-driver ...
/task-driver ...
```

例如：

```text
tdr- 先和我讨论登录页按钮错位的修复范围，再形成计划并执行
```

## 行为原则

- 每轮只推进一个需要用户决定的问题。
- 模型判断讨论是否充分，但只能由用户决定是否结束讨论并进入下一环节。
- 未经确认的建议不能写成默认技术决策。
- 顶层计划事项对应已确认的需求或问题。
- 每个计划事项必须展示“已确认”和“实施方案”。
- 计划确认后先讨论最终结果，不直接生成验收目标。
- 最终验收目标只表达必要终极结果，不扩散成施工检查清单。
- 最终验收目标确认后必须原样创建目标或维护轻量目标。
- 辅助测试不能替代用户确认的最终验收目标。
- 不自动新建分支、提交、推送、合并或真实调用付费服务。
- 未达成或无法确认达标时先分类；只有可修复问题进入最多 3 次、每次改变策略的修复循环。

## 最短完整流程

```text
用户：tdr- 先讨论登录按钮错位和移动端溢出，再给计划。只修位置和溢出，不统一其他按钮样式。

助手：逐项讨论两个问题，记录用户决定；判断内容充分后，让用户选择继续讨论或生成计划。

用户：生成计划。

助手：输出两个对应计划事项，每项展示“已确认”和“实施方案”，等待用户确认。

用户：确认。

助手：逐项讨论两个问题的最终完成状态；判断充分后，让用户选择继续讨论或生成目标。

用户：生成目标。

助手：输出两个必要终极目标；用户确认后原样创建目标并执行、自检和修复。
```

## 计划展示

计划默认按已确认事项纵向展示：

```markdown
## 1. {计划事项}

### 已确认
- {关键决策和边界}

### 预期变化
{复杂任务按需展示}

### 实施方案
{核心机制、作用边界和关键约束}

### 涉及位置
{复杂任务按需展示}

### 内部验证
{复杂任务按需展示}
```

简单任务可以省略辅助板块，但不能省略“已确认”和“实施方案”。文件、测试、文档和版本工作不膨胀成新的顶层计划事项。

## 最终验收目标

最终验收目标以已确认需求或问题为主轴：

- 原则上一项独立需求或问题对应一个最终目标。
- 结果完全重叠时可以合并，并说明来源和原因。
- 一个事项包含多个可独立成败的最终结果时可以拆分，并说明来源和原因。
- 测试、脚本、文档和版本默认是内部证据，不是终极目标。
- 用户确认后的目标原样进入目标模式。

## 宿主目标工具

`TaskCreate`、`TaskList` 或 `TaskUpdate` 等 deferred tool 的 schema 未加载时，必须先通过 `ToolSearch` 精确加载，再按参数定义调用；出现 `Invalid tool parameters` 后不得继续猜测参数。

宿主任务只映射最终验收目标，不映射实施步骤。创建后保存任务 ID，开始执行时同步为执行中，确认达标后立即同步为已完成；中断恢复时先读取和复用已有任务，禁止重复创建。

## 任务规模

- 同一个业务目标、权限和回滚方式一致：可以一次推进。
- 同一个目标但跨多轮、依赖或风险较多：维护一份轻量计划摘要。
- 多个可独立交付目标、不同授权或不同发布回滚周期：建议拆分，并由用户确认第一批范围。

任务规模不按固定文件数或目标数判断。详细规则见 [常见问题](references/faq.md#怎样判断任务是否应该拆分) 和 [完整协议](SKILL.md#任务规模与拆分边界)。

## 权限边界

新建分支、提交、推送、合并、删除或覆盖重要文件、发布部署、改生产配置、真实调用外部服务或产生费用，都必须先获得用户明确授权。

## 需要用户验收时

Agent 必须先完成自己能够执行的检查。确实依赖人的主观感受、特定设备、私有账号或无法访问的外部环境时，目标暂不判定达标，并向用户提供：

- 无法由 Agent 可靠验证的原因。
- 可以直接照做的编号验收步骤。
- 每一步应看到的结果和最终通过条件。
- 用户需要回复的结果或证据。

不能只说“请自行检查”。用户反馈不通过时必须继续修复；必要验收没有结果时不能完成目标。

## 轻量计划摘要

普通任务不写摘要。大任务、超大任务或长执行需要防止上下文丢失时，只维护一份：

```text
.task-driver/{任务标题}-{YYYYMMDDHHmmss}-计划摘要.md
```
