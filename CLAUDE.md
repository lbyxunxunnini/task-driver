# 任务驱动项目规则

## 项目信息

- **名称**：任务驱动
- **版本**：v0.9.3
- **定位**：低认知负担的复杂任务推进 skill
- **GitHub**：[lbyxunxunnini/task-driver](https://github.com/lbyxunxunnini/task-driver)
- **父级规则**：[agent设计/CLAUDE.md](../CLAUDE.md)

## 当前核心链路

```text
仔细询问并逐项讨论
-> 输出计划
-> 讨论并确认计划
-> 讨论验收目标
-> 输出并确认验收目标
-> 创建目标并执行
-> 未达标或无法确认达标时先分类
-> 可修复问题打回修复，最多 3 次；其他异常进入对应路径
```

## 开发规则

- 用户可见流程必须使用中文自然语言。
- 不再恢复旧的重流程和中间记录体系。
- 不再默认写中间文档；只有用户明确要求时才写。
- 模型判断讨论是否充分，但必须由用户决定是否输出计划或最终验收目标。
- 顶层计划事项必须对应已确认的需求或问题，并展示“已确认”和“实施方案”。
- 最终验收目标必须先逐项讨论，再单独输出和确认。
- 最终验收目标只表达必要终极结果，不按文件、测试或施工步骤扩散。
- 工程测试只能作为辅助检查，不能替代最终验收目标。
- 最终验收目标确认后必须原样创建目标或维护轻量目标。
- deferred 目标工具必须先加载 schema 再调用；宿主任务必须保存 ID、同步执行与完成状态，并在中断恢复时复用。
- 未达成最终验收目标必须打回修复，全部达成后才能完成目标。
- Agent 无法可靠验证目标时必须通知用户验收，提供具体步骤、通过条件和反馈要求；结果返回前不得标记完成。
- 任务规模必须按业务目标、权限、风险、交付和回滚关系判断；建议拆分时必须说明依据并让用户确认本轮范围。
- 异常必须先分为可修复、需要用户验收、需要用户授权或决定、受阻；只有可修复问题进入最多 3 次的修复循环。
- 每次重试必须改变实施策略或修复内容；相同原因和相同方法不得机械重试。
- 异常通知必须说明当前结果、具体原因、已完成内容和下一步；需要用户操作时必须提供步骤与通过条件。
- README 只承担快速判断和文档导航；Quick Start、FAQ、场景指南和完整协议按需读取，不大段互相复制。
- 大任务写入必须先骨架后分块，避免一次性写入过大内容。
- 一个项目的一个任务最多只维护一份 `.task-driver/{任务标题}-{时间戳}-计划摘要.md`。
- 大任务、超大任务或长执行由模型自主判断必须写摘要；普通任务默认不写。
- 不得默认新建分支、提交、推送或合并。
- 修改协议后必须运行 `scripts/check-contracts.sh` 和 `git diff --check`。

## 版本管理

- 版本号同步：`SKILL.md` / `VERSION.md` / `.skillhub.json` / `README.md` / `CHANGELOG.md`。
- 版本对齐基准：本地 `VERSION.md` 文件。
- 发布前运行 `bash scripts/release_checks/metadata.sh`。

## Git 约束

- 提交信息前缀：`feat:` / `fix:` / `docs:` / `refactor:`
- 独立推送到 GitHub，不与其他子项目共享 git 历史。
